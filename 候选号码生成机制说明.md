# 候选号码生成机制说明

## 问题

用户疑问：候选号码不是通过尾号预测组件吗，当前为什么还是候选号码评分？

## 答案

候选号码的生成确实是**尾号预测驱动**的，但评分系统仍然有重要作用。

## 完整流程

```
尾号预测组件（11步）
      ↓
预测Top5尾号 + 邻号(±1)
      ↓
所有尾号匹配的球入池（策略1）
      ↓
源号码入池（策略2）
      ↓
区间保底补齐（策略3）
      ↓
按分数补齐到25/30个（策略4）
      ↓
评分排序 → 最终候选池
```

## 详细说明

### 1. 尾号预测驱动（核心）

```javascript
// 获取预测尾号（Top5）
const predictedTailSet = new Set();
predictedTails.slice(0, 5).forEach(([tail]) => {
  predictedTailSet.add(tail);           // 预测尾号
  predictedTailSet.add((tail + 1) % 10); // 邻号+1
  predictedTailSet.add((tail + 9) % 10); // 邻号-1
});

// 策略1：所有预测尾号（含邻号）的球入池
for (const c of candidates) {
  const t = c.number % 10;
  if (predictedTailSet.has(t)) {
    pool.push(c);
  }
}
```

**核心思想**：预测尾号 → 所有尾号匹配的球入池

### 2. 评分系统的作用

评分系统**不是**用来决定哪些号码进入候选池，而是用来：

#### 2.1 排序优先级
- 在尾号匹配的号码中，高分号码优先
- 确保最有可能命中的号码排在前面

#### 2.2 区间平衡
```javascript
// 策略3：区间保底——按分数排序补齐缺失区间
if (zoneCount[z] < minIv[z]) {
  pool.push(c); // 补齐缺失区间
}
```

#### 2.3 候选池大小控制
```javascript
// 策略4：按分数补齐到25个
const MIN_POOL_SIZE = 25;
for (const c of candidates) {
  if (pool.length >= MIN_POOL_SIZE) break;
  if (!seen.has(c.number)) {
    pool.push(c);
  }
}
```

## 为什么需要评分？

### 1. 尾号预测的局限性
- 尾号预测只能预测**哪些尾号可能出现**
- 但同一个尾号可能有多个号码（如尾号1：1, 11, 21, 31）
- 需要评分来区分同一尾号内的号码优先级

### 2. 区间平衡需求
- 尾号预测可能导致某些区间号码过多或过少
- 评分系统确保三个区间都有足够的候选号码

### 3. 候选池大小固定
- 系统需要固定大小的候选池（25或30个）
- 评分系统用于在尾号匹配的号码中选择最优的25/30个

## 实际效果

### 候选池构成
1. **主要部分**：尾号预测匹配的号码（约60-70%）
2. **补充部分**：区间保底补齐的号码（约20-30%）
3. **填充部分**：按分数补齐到指定大小（约10%）

### 评分维度（16维度）
1. 偏移评分
2. 尾号关联
3. 区间稳定尾号加成
4. 尾号关联性
5. +10期趋势
6. 桥接分数
7. 等差分数
8. 热号分数
9. 极端期加成
10. 连号附近奖励
11. 区间平衡
12. 奇偶平衡
13. 和值贡献
14. 历史频率
15. 首位球预测
16. 特殊号码关联

## 总结

| 组件 | 作用 | 决定什么 |
|------|------|----------|
| 尾号预测组件 | 预测哪些尾号可能出现 | 候选池的**主体构成** |
| 评分系统 | 对候选号码排序 | 候选池的**优先级和补充** |

**核心逻辑**：
1. 尾号预测决定**哪些号码有资格进入候选池**
2. 评分系统决定**这些号码的优先级顺序**
3. 两者结合确保候选池既符合尾号预测，又满足区间平衡和大小要求
